# 1인 웹사이트 제작 에이전트 시스템 계획서 검토 보고서

> 대상 문서: [agency-ai-agent-plan.md](/Volumes/sub_ssd/Works/Antigravity/WebStart/agency-ai-agent-plan.md)  
> 작성일: 2026-04-02  
> 목적: 문제점, 개선안, 실행 계획을 한 번에 정리한 검토 보고서

## 1. 요약

원본 계획서는 에이전트 역할 분리와 작업 흐름이 잘 잡혀 있다. 다만 실제 운영에 들어가면 다음 문제가 발생할 가능성이 높다.

- FE/BE 병렬 작업 전에 공통 계약이 고정되지 않아 통합 충돌이 생길 수 있다.
- QA 통과 기준이 약해 배포 승인 품질을 충분히 보장하지 못한다.
- 범위 변경이 발생했을 때 PM으로 되돌리는 통제 흐름이 없다.
- DevOps 문서가 로컬 절대경로에 의존해 이식성이 떨어진다.
- Claude.ai Project와 CLAUDE.md 사이의 기준 문서가 불명확하다.

결론적으로, 현재 문서는 역할표로는 충분하지만 실행 문서로는 보강이 필요하다.

## 3. 개선 계획

### 3.1 구조 개선

1. **요구사항 수집 단계 강화**
   - 클라이언트 브리프에 필수 입력값을 고정한다.
   - 예산, 일정, 필수 기능, 제외 범위를 명시한다.
   - 승인 전에는 디자인과 개발 작업을 시작하지 않는다.

2. **Contract Freeze 단계 추가**
   - 산출물: 사이트맵, 핵심 사용자 흐름, API 계약, 데이터 모델, 상태 정의
   - 목적: FE/BE가 같은 기준으로 병렬 작업하도록 만드는 것
   - 완료 조건: PM, Design, FE, BE가 동일 문서를 승인

3. **QA 승인 기준 강화**
   - 기능 테스트
   - 접근성 테스트
   - 성능 기준
   - SEO 기준
   - 배포 후 smoke test
   - 롤백 가능성 확인

4. **변경 관리 규칙 추가**
   - 새 요구사항이 생기면 PM으로 되돌린다.
   - 디자인 영향이 있으면 디자인 시스템을 먼저 갱신한다.
   - 구현 영향이 있으면 FE/BE 계약을 다시 맞춘다.

### 3.2 문서 개선

1. 공통 규칙에 `single source of truth` 원칙을 추가한다.
2. 각 에이전트 프롬프트에 입력, 출력, 금지 항목을 더 명확히 분리한다.
3. DevOps 섹션의 로컬 경로 의존성을 제거한다.
4. 배포 승인 조건에 QA 완료 및 smoke test 성공을 명시한다.

### 3.3 운영 개선

1. 새 프로젝트 시작 시 `PM -> Design -> Contract Freeze -> FE/BE -> QA -> DevOps` 순서를 기본값으로 둔다.
2. 병렬 작업은 `Contract Freeze` 이후에만 허용한다.
3. QA 실패 시 수정 책임을 FE/BE로만 넘기지 말고, 실패 원인이 요구사항인지 구현인지 먼저 분류한다.
4. 배포 전에는 최소 1회 production-like smoke test를 수행한다.

## 5. 최종 판단

현재 계획서는 시작점으로는 좋지만, 그대로 운영하면 재작업과 기준 불일치가 생길 가능성이 높다.

가장 먼저 해야 할 일은 병렬 개발 전에 공통 계약을 고정하는 것이다. 그 다음 QA 기준과 변경 관리 규칙을 잠그면, 문서가 실제 작업 흐름을 제대로 통제할 수 있다.
